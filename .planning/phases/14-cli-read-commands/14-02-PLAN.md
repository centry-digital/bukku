**`formatTable(rows: Record<string, unknown>[], columns: Column[]): string`** function:
1. Write header row: each column header padded to width, separated by spaces (2-space gap).
2. Write separator row: dashes for each column width.
3. For each data row: extract value by key, convert to string, truncate to width (add "..." if truncated), pad to width respecting alignment.
4. Return the full string.

**Value extraction:** Get value from record by key. Handle nested keys with dot notation if needed, but for now simple top-level access is sufficient. Convert null/undefined to empty string. Convert numbers to string. Convert booleans to "yes"/"no".

**`outputTable(data: unknown[], entityName?: string): void`** (exported function):
1. Lookup columns: `TABLE_COLUMNS[entityName] ?? DEFAULT_COLUMNS`
2. Cast data to `Record<string, unknown>[]`
3. Call `formatTable` and write result to stdout via `process.stdout.write`
4. If data is empty, write "(no results)\n" to stdout

Keep the function signature compatible with what Plan 14-01's factory calls: `outputTable(items, config.entity)`.

**Part A: Create `packages/cli/src/commands/custom/reference-data.ts`**

Export a function:
```typescript
export function registerReferenceDataCommands(program: Command): void
```

This creates 10 reference data list commands. These use the POST `/v2/lists` API, not standard GET CRUD endpoints.

Group the reference data commands under the `accounting` group command (since tax codes, currencies, accounts, terms, etc. are accounting reference data). The command names should be:
- `bukku accounting tax-codes` (list tax codes)
- `bukku accounting currencies` (list currencies)
- `bukku accounting payment-methods` (list payment methods)
- `bukku accounting terms` (list payment terms)
- `bukku accounting accounts-ref` (list accounts — use "accounts-ref" to avoid conflict with the CRUD "accounts" resource subcommand; this is the reference data quick lookup)
- `bukku accounting price-levels` (list price levels)
- `bukku accounting countries` (list countries)
- `bukku accounting classification-codes` (list classification codes)
- `bukku accounting numberings` (list numbering schemes)
- `bukku accounting states` (list geographic states)

Actually, reconsider grouping: some reference data is more "settings" than "accounting". Use a dedicated `ref-data` top-level group instead to avoid cluttering accounting:

```
bukku ref-data tax-codes [--format json|table]
bukku ref-data currencies [--format json|table]
bukku ref-data payment-methods [--format json|table]
bukku ref-data terms [--format json|table]
bukku ref-data accounts [--format json|table]
bukku ref-data price-levels [--format json|table]
bukku ref-data countries [--format json|table]
bukku ref-data classification-codes [--format json|table]
bukku ref-data numberings [--format json|table]
bukku ref-data states [--format json|table]
```

Implementation:
1. Define the same `REFERENCE_TYPES` array as in the MCP reference-data.ts (type, commandName, description).
2. Create `program.command('ref-data').description('Reference data lookups (tax codes, currencies, etc.)')`.
3. For each reference type, add a subcommand that:
   - Accepts `--format <json|table>` (default: json)
   - Uses `withAuth` wrapper
   - Calls `client.post('/v2/lists', { lists: [type], params: [] })`
   - Outputs result with `outputJson` or `outputTable`
4. No caching needed for CLI (short-lived process, unlike MCP server).

**Part B: Create `packages/cli/src/commands/custom/search-accounts.ts`**

Export a function:
```typescript
export function registerSearchAccountsCommand(program: Command): void
```

This adds `bukku accounting search-accounts` command:
- Find the `accounting` group command on program
- Add `search-accounts` subcommand with options:
  - `--search <text>` (search by name/code)
  - `--category <cat>` (choices: assets, liabilities, equity, income, expenses)
  - `--is-archived` (boolean flag, default false)
  - `--sort-by <field>` (choices: code, name, balance)

  - Build query params, converting `--is-archived` flag to string "true"/"false", `--limit` to `page_size`
  - Call `client.get('/accounts', params)`
  - Output with outputJson or outputTable (entity name "account" for table columns)
