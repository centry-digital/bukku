  <action>
Create `packages/cli/src/commands/factory.ts` that exports a single function:

```typescript
export function registerEntityCommands(program: Command): void
```

This function:

1. **Build group command map.** Iterate `allEntityConfigs` from core. For each config with a `cliGroup`, find or create the matching top-level Commander subcommand on `program`. Group names are the `cliGroup` values: "sales", "purchases", "banking", "contacts", "products", "accounting", "files", "control-panel".

2. **Derive resource subcommand name** from `config.entity` by stripping the group prefix. Examples:
   - entity="sales-invoice", cliGroup="sales" -> resource subcommand = "invoices"
   - entity="bank-money-in", cliGroup="banking" -> resource subcommand = "money-in"
   - entity="contact", cliGroup="contacts" -> resource subcommand = "contacts" (same as group, entity has no prefix to strip)
   - entity="product-bundle", cliGroup="products" -> resource subcommand = "bundles"
   - entity="tag-group", cliGroup="control-panel" -> resource subcommand = "tag-groups"
   - entity="delivery-order", cliGroup="sales" -> resource subcommand = "delivery-orders"
   - entity="goods-received-note", cliGroup="purchases" -> resource subcommand = "goods-received-notes"

   Strategy: Build a mapping function `deriveResourceName(entity: string, cliGroup: string): string`. Logic:
   - For single-word groups (sales, purchases, banking, contacts, products, files): if entity starts with a known prefix for that group, strip it and pluralize; otherwise pluralize the entity.
   - The prefix mapping: sales->"sales-", purchases->"purchase-", banking->"bank-", contacts->"contact", products->"product", files->"file", accounting->"", control-panel->""
   - Special cases to handle: "delivery-order" under sales (no "sales-" prefix, use "delivery-orders"), "goods-received-note" under purchases (no "purchase-" prefix, use "goods-received-notes"), "journal-entry" under accounting (use "journal-entries"), "account" under accounting (use "accounts")
   - For control-panel group: use entity name pluralized directly (location->"locations", tag->"tags", tag-group->"tag-groups")

   Rather than complex prefix logic, use a simple explicit map for the ~27 entities. This is more maintainable:
   ```typescript
   const RESOURCE_NAME_MAP: Record<string, string> = {
     'sales-invoice': 'invoices',
     'sales-quote': 'quotes',

     'sales-credit-note': 'credit-notes',

     'purchase-order': 'orders',
     'purchase-credit-note': 'credit-notes',
     'purchase-payment': 'payments',
     'purchase-refund': 'refunds',
     'goods-received-note': 'goods-received-notes',
     'bank-money-in': 'money-in',
     'bank-money-out': 'money-out',
     'bank-transfer': 'transfers',
     'contact': 'contacts',
     'contact-group': 'groups',
     'product': 'products',
     'product-bundle': 'bundles',
     'product-group': 'groups',
     'journal-entry': 'journal-entries',
     'account': 'accounts',
     'file': 'files',

   };
   ```

3. **For each config with "list" in operations**, add a `list` subcommand on the resource command:
   - Standard options: `--limit <n>` (alias for page_size), `--page <n>`, `--search <text>`, `--format <format>` (choices: json, table; default: json), `--date-from <date>`, `--date-to <date>`, `--status <status>`, `--sort-by <field>`, `--sort-dir <dir>` (choices: asc, desc)
   - `--all` boolean flag for auto-pagination
   - Entity-specific filter options from `config.listFilters` — add each as `--<filter-name> <value>` (converting underscores to hyphens in the flag name)
   - Handler wrapped with `withAuth`. Inside handler:
     - Build query params object from opts, converting `limit` to `page_size`, converting hyphenated flag names back to underscore param names
     - If `--all` flag: loop fetching pages, collecting results. Use `paging.total` and `paging.per_page` to determine total pages. Write progress to stderr: `Fetching page N/M...`. Concatenate all items arrays.
     - Call `client.get(config.apiBasePath, params)`
     - Response is `{ paging: {...}, [config.pluralKey]: [...] }`
     - If `--format table`: call `outputTable(items, config.entity)` (from `../output/table.ts`)
     - Otherwise: call `outputJson(data)` (full response with paging for single-page, or just the items array for --all)

4. **For each config with "get" in operations**, add a `get <id>` subcommand:
   - Argument: `<id>` (required, parsed as integer)
   - Options: `--format <format>` (choices: json, table; default: json)
   - Handler wrapped with `withAuth`:
     - Call `client.get(\`${config.apiBasePath}/${id}\`)`
     - Response is `{ [config.singularKey]: {...} }`
     - If `--format table`: call `outputTable([item], config.entity)` (wrap single item in array)
     - Otherwise: call `outputJson(data)` (full response)

5. **Import outputTable** from `../output/table.ts` — the current placeholder will fall back to JSON, which is fine for now (Plan 14-02 replaces it).

Key implementation details:
- Use `program.commands.find(c => c.name() === groupName)` to find existing group commands, or `program.command(groupName)` to create new ones. Since index.ts already creates the group commands, the factory should find them. But index.ts uses "settings" while configs use "control-panel" — the factory should create "control-panel" if it doesn't exist already.
- Resource commands are subcommands of group commands: `groupCmd.command(resourceName).description(...)`.
- `list` and `get` are subcommands of resource commands: `resourceCmd.command('list')`, `resourceCmd.command('get')`.
- For the `--all` pagination loop, use a reasonable default page_size of 100 if not specified.
- Parse `<id>` argument with `parseInt` and validate it's a positive integer before calling API.
