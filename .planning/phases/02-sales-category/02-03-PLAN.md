<objective>
Close GAP-01: Add business-rule context to MCP tool descriptions so the LLM knows which
recovery options are viable when operations fail due to status constraints.

Purpose: When a user tries to delete a ready-status invoice, the LLM currently suggests
invalid recovery options (e.g., "revert to draft") because tool descriptions carry no
business-rule context. By embedding status lifecycle rules into tool descriptions, the
LLM will only suggest viable alternatives (e.g., void the invoice).

Output: Updated CrudEntityConfig type, enriched factory-generated tool descriptions,
and business rules declared in all 7 sales entity configs.
</objective>

Add a `businessRules` object to each entity config. The rules describe Bukku's status lifecycle and delete constraints so the LLM knows what is and is not possible.

**Bukku status lifecycle (common across sales transaction types):**
All sales transaction types in Bukku follow the same status pattern:
- Statuses: draft, ready (also called "approved"/"posted"), void
- Draft can transition to: ready
- Ready can transition to: void
- Void is terminal (no further transitions)
- Only draft-status documents can be deleted
- Ready and void documents cannot be deleted and cannot revert to draft

Apply the following `businessRules` to each config. The wording is intentionally LLM-facing (concise, declarative, no jargon):

**sales-invoice.ts:**
```ts

},
```
