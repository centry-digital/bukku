<task type="auto">
  <name>Task 1: Implement CRUD factory pattern</name>
  <files>src/tools/factory.ts</files>
  <action>
Create the CRUD factory that generates MCP tool registrations from entity configuration objects.

The factory function `registerCrudTools(server: McpServer, client: BukkuClient, config: CrudEntityConfig)` should generate up to 6 tools per entity:

**list-{entity}s** (if 'list' in operations):
- Name: `list-{entity}s` (e.g., `list-sales-invoices`). Per locked decision: kebab-case, no prefix.
- Description: `List ${description}s with optional filters` (e.g., "List sales invoices with optional filters")
- Input schema (Zod): page (number, optional), page_size (number, optional), search (string, optional), date_from (string, optional), date_to (string, optional), status (string, optional), sort_by (string, optional), sort_dir (enum 'asc'|'desc', optional). Plus any extra filters from config.listFilters.
- Handler: calls `client.get(config.apiBasePath, params)`, returns JSON as text content.
- Error handling: catch block calls `transformHttpError()`, returns MCP error response.

**get-{entity}** (if 'get' in operations):
- Name: `get-{entity}` (e.g., `get-sales-invoice`)
- Description: `Get a ${description} by ID`
- Input schema: id (number, required, describe as "The {description} ID")
- Handler: calls `client.get(${config.apiBasePath}/${id})`
- Error handling: same pattern

**create-{entity}** (if 'create' in operations):
- Name: `create-{entity}` (e.g., `create-sales-invoice`)
- Description: `Create a new ${description}`
- Input schema: data (object, passthrough -- accepts any fields to forward to Bukku API). Use `z.record(z.unknown())` or `z.object({}).passthrough()` to allow flexible input since each entity has different create fields. Bukku API handles field validation.
- Handler: calls `client.post(config.apiBasePath, data)`
- Error handling: same pattern

**update-{entity}** (if 'update' in operations):
- Name: `update-{entity}` (e.g., `update-sales-invoice`)
- Description: `Update an existing ${description}`
- Input schema: id (number, required), data (object, passthrough)
- Handler: calls `client.put(${config.apiBasePath}/${id}, data)`

**delete-{entity}** (if 'delete' in operations):
- Name: `delete-{entity}` (e.g., `delete-sales-invoice`)
- Description: `Delete a ${description}`
- Input schema: id (number, required)
- Handler: calls `client.delete(${config.apiBasePath}/${id})`

**update-{entity}-status** (if config.hasStatusUpdate is true):
- Name: `update-{entity}-status` (e.g., `update-sales-invoice-status`). Per locked decision: dedicated status tools.
- Description: `Update the status of a ${description}`
- Input schema: id (number, required), status (string, required, describe as "New status value")
- Handler: calls `client.patch(${config.apiBasePath}/${id}, { status })`

For ALL handlers:
- Wrap the API call in try/catch
- On success: return `{ content: [{ type: 'text', text: JSON.stringify(result, null, 2) }] }`
- On caught error: use `transformHttpError()` for HTTP errors and `transformNetworkError()` for network errors
- NEVER use console.log -- use the logger for any debug output

Return the count of tools registered (for logging).
