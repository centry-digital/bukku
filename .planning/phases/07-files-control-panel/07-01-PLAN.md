  <name>Task 1: Add postMultipart method to BukkuClient and create file entity config</name>
  <files>src/client/bukku-client.ts, src/tools/configs/file.ts</files>
  <action>
1. **Add postMultipart method to BukkuClient** (`src/client/bukku-client.ts`):
   - Add imports at top: `import { readFile } from "node:fs/promises";` and `import { basename, extname } from "node:path";`
   - Add a `getMimeType` private helper method that maps common extensions to MIME types: .pdf -> application/pdf, .png -> image/png, .jpg/.jpeg -> image/jpeg, .gif -> image/gif, .txt -> text/plain, .csv -> text/csv, .json -> application/json, .xml -> application/xml, .zip -> application/zip. Returns null for unknown extensions.
   - Add `async postMultipart(path: string, filePath: string): Promise<unknown>` method:
     - Read file with `await readFile(filePath)`
     - Get filename via `basename(filePath)`, extension via `extname(filePath)`
     - Determine MIME type via getMimeType helper, fallback to "application/octet-stream"
     - Create `new File([fileBuffer], fileName, { type: mimeType })`
     - Create `new FormData()`, append file as `form.append("file", file)` — the API expects the field name "file"
     - CRITICAL: Call `this.getHeaders(false)` to get auth headers WITHOUT Content-Type. Fetch MUST set Content-Type with multipart boundary automatically. Manually setting Content-Type breaks the upload.
     - Execute `fetch(url, { method: "POST", headers, body: form })`
     - On `!response.ok`, throw response (same pattern as other methods)
     - Return `response.json()`
   - The `getHeaders` method already accepts a boolean parameter and only adds Content-Type when true, so `getHeaders(false)` is correct — no changes needed there.

2. **Create file entity config** (`src/tools/configs/file.ts`):
   - Import `CrudEntityConfig` from `../../types/bukku.js`
   - Export `fileConfig: CrudEntityConfig` with:
     - entity: "file"
     - apiBasePath: "/files"
     - singularKey: "file"
     - pluralKey: "files"
     - description: "file. Files are typically attached to sales and purchase transactions using file_ids arrays. Use upload-file to add new files."
     - operations: ["list", "get"] — NO create/update/delete (create handled by custom upload tool, API has no update/delete for files)
     - hasStatusUpdate: false
     - listFilters: [] — API spec shows no query parameters for GET /files (no pagination, no filters)
   - Add JSDoc noting: No delete endpoint in API (files are permanent). No update endpoint (files are immutable). List returns ALL files (no pagination support).
